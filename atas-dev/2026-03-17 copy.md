# Ata de Reunião - Modelagem do Sistema de Bingo
Data: 2026-03-17

## 1. Decisão Geral

Devido à limitação de 64 bits do banco de dados, a estratégia de uso de bitmask foi descartada para esta fase do projeto.

Fica definido o uso de:
- JSON como fonte de verdade dos números
- Hash para validação de unicidade e performance

---

## 2. Regras de Negócio Definidas

- Uma paróquia é a entidade central do sistema
- Uma paróquia possui vários usuários
- Um usuário-comum possui várias cartelas
- Uma paróquia possui vários sorteios
- Um sorteio possui apenas uma cartela vencedora
- Regra de vitória:

cartela ⊆ sorteio

- O prêmio será dividido entre todos os vencedores

---

## 3. Banco de Dados

### 3.1 Ajustes obrigatórios

#### Tabela: usuarios
- paroquia_id deve ser NOT NULL

#### Tabela: cartelas
Campos:
- id
- usuario_id (NOT NULL)
- sorteio_id (NOT NULL)
- numeros (JSON NOT NULL)
- hash (TEXT NOT NULL)

Constraints:
- UNIQUE (sorteio_id, hash)

#### Tabela: sorteios
Campos:
- id
- paroquia_id (NOT NULL)
- numeros_sorteados (JSON)
- cartela_vencedora_id (UNIQUE)

---

## 4. Backend (FastAPI)

### 4.1 Validações obrigatórias

Cartelas:
- números entre 1 e 75
- sem repetição
- ordenados antes de gerar hash

Sorteios:
- números entre 1 e 75
- sem repetição
- manter ordem de sorteio (NÃO ordenar)

### 4.2 Geração de Hash

- Gerar hash (SHA-256) a partir do JSON ordenado da cartela

### 4.3 Verificação de vencedores

- Ordenar números do sorteio (cópia)
- Comparar usando lógica de subconjunto (set)

---

## 5. Frontend

### 5.1 Criação de cartelas

- Garantir envio dos números:
  - ordenados (crescente)
  - sem repetição

### 5.2 Comunicação com backend

- Enviar cartelas como JSON
- Não enviar hash (gerado no backend)

---

## 6. Observações Técnicas

- JSON é a fonte de verdade
- Hash é usado para performance e unicidade
- Não utilizar comparação direta de JSON no banco
- Não utilizar bitmask nesta fase
